# Post-1.0 Hardening Arc — Design

**Date:** 2026-05-29
**Status:** Approved (brainstorming)

## Context

The original 14-phase build arc is complete (final phase 14, Wikidata GraphQL, at
`20117ac`). A round of hands-on spelunking surfaced one critical data-loss bug, two
small UX/registry papercuts, and three genuinely missing features. This document
defines a post-1.0 hardening arc: **two immediate one-off fixes** plus **three focused
phases (15–17)**, ordered so the critical fix lands first.

Layer boundaries (`core ↔ api ↔ frontend ↔ mcp`) are unchanged; every item is additive
within existing layers, so no ADR is required. `make check` must be green before each
commit.

## Findings (the spelunking that motivated this)

1. **Variables** are defined only in `.drummer/environments/<name>.yaml`
   (`name:` + a `variables:` map; resolved by `drummer/core/variables.py`). The UI
   (`Sidebar.tsx`) shows a read-only environment selector — there is no way to view or
   edit variables. Setting `{{base_url}}` currently requires hand-editing YAML.
2. **Adding environments** is not possible from the UI and barely from the API:
   `environments.py` has GET + PUT, but PUT 404s if the file does not already exist.
   No create, no delete.
3. **"Demo Project"** is the e2e fixture `tests/e2e/fixtures/demo-project/` (its
   `.drummer/project.yaml` is named "Demo Project"). It appears in the workspace
   switcher because it got registered as an *external* workspace in the user's
   `~/.drummer` registry (via a `make dev PROJECT=…` or an e2e run) and nothing ever
   forgets it.
4. **Raw response tab shows the body twice.** `RawViewer.tsx` renders a hexdump table
   (offset / hex / ASCII) *and* a separate full-text `<pre>` panel on the right.
5. **Saving is a data-loss bug, not just wonky:**
   - Blank-screen crash: the PUT route returns a `RequestSummary`
     (`{path, name, method, url}`), but the frontend (`requests.ts`) types the response
     as `RequestDetail` and calls `markSaved(result)`; `result.frontmatter` is
     `undefined`, so the request pane crashes to a white screen.
   - Silent data loss: both the save route and the frontend payload carry only
     `name, method, url, headers, body`. On every save, `params`, `auth`, `cookies`,
     pre/post `scripts`, `graphql`, and `encoding` are reset to defaults **on disk**.
   - No visible Save affordance — only a dirty dot in the tree and cmd-s.
6. **No way to add requests** from the UI. `POST /requests` exists on the backend, but
   there is no `useCreateRequest` hook and no button; no rename/move/delete in the tree
   either.
7. **No sent-request view.** The engine builds a `ResolvedRequest` (final URL,
   substituted params/headers, variable set, unresolved-variable warnings), but the
   response side never surfaces any of it.
8. **`:8000` also serving the UI is intentional, not a clone.** `drummer/api/app.py`
   mounts the compiled React bundle (`/assets` via `StaticFiles`) with an SPA fallback
   serving `index.html`, so one `drummer` command provides API + UI on one port. The
   Vite `:5173` server is dev-only (`make dev`). No code change required; document only.

## Immediate one-off fixes (not phased — small, do first)

### F1 — Raw tab redundancy
`RawViewer.tsx`: remove the right-hand `<pre>` body panel; keep the hexdump (its ASCII
column already shows the text). Raw becomes a pure hexdump.

### F2 — Forget external workspaces
- Add a way to remove/forget an external workspace from the `~/.drummer` registry, so
  the "Demo Project" fixture can be cleared.
- Stop test fixtures from auto-registering into the user's real registry: tests should
  use an isolated registry path so running e2e/dev never pollutes `~/.drummer`.

## Phase 15 — Request editing & CRUD *(contains the critical save fix — first)*

- **15a (critical):** Stop the save data-loss. The save route + frontend payload must
  round-trip the **full** `RequestFrontmatter` (params, auth, cookies, pre/post
  scripts, graphql, encoding) — not just name/method/url/headers/body.
- **15b (critical):** Fix the blank-screen crash. The PUT route returns a full
  `RequestDetail` (not `RequestSummary`); the frontend type and `markSaved` must match.
- **15c:** Visible Save affordance — a Save button (enabled only when dirty) alongside
  the existing dirty dot; keep cmd-s working.
- **15d:** Request CRUD in the tree — new request, rename, delete. Move/folders are
  optional and decided during plan-writing. Wires up the existing `POST /requests` plus
  a new `useCreateRequest` hook.

## Phase 16 — Environment & variable editor

- Backend: add **create** (POST) and **delete** for environments (today PUT 404s if the
  environment file is absent).
- Frontend: a variable-table editor (reuse the existing `KeyValueTable` pattern) to
  add/edit/remove variables per environment, plus create/rename/delete environments.
  Turns the read-only selector into real management.

## Phase 17 — Sent-request inspector

- Surface the engine's already-computed `ResolvedRequest` on the response side: a "Sent"
  view showing the final URL, substituted params/headers, and the variable set used.
- Surface **unresolved-variable warnings** (already collected by the engine) so a stray
  `{{typo}}` is visible instead of silently sent literally.

## Out of scope

- Documenting the dev-vs-prod server story (the `:8000`/`:5173` explanation) beyond an
  optional note — no behavior change.
- Move/folders in the request tree unless chosen during Phase 15 plan-writing.
- Any change to layer boundaries (no ADR triggered).

## Conventions

- Each phase gets its own implementation plan in `docs/superpowers/plans/` when work
  begins; this document is the arc-level spec.
- `make check` (ruff + pyright + pytest) green before each commit; no suppression
  comments.
